# RT-VC: Deep Analysis & Suitability for rvc-web

**Paper:** RT-VC: Real-Time Zero-Shot Voice Conversion with Speech Articulatory Coding  
**Authors:** Yisi Liu et al., UC Berkeley (ACL 2025)  
**Repo:** `~/code/RT-VC` — inference-only, no training code

---

## What it is in plain terms

RT-VC converts someone's voice to sound like a different speaker **in real time**, with **zero prior exposure to either speaker**. You don't fine-tune anything for a new voice — you just give it a short clip of the target speaker and it adapts on the fly.

It achieves **61.4ms end-to-end CPU latency** (measured on Apple M3), which is below the perceptual threshold for "live" feel in a conversation.

The key idea that makes this work is using **articulatory features** (how your mouth, tongue, and jaw move) as the intermediate representation, rather than the abstract neural embeddings most VC systems use. This is a principled disentanglement: articulation encodes *what you're saying* (content), while a speaker embedding encodes *who you sound like* (identity). These two things never mix during conversion.

## Architecture: Four Components

```
Audio in (16kHz)
    │
    ├──► [1] Source Extractor     ──► f0 (pitch), periodicity, loudness  @200Hz
    │         (Causal Conv on mel)
    │
    ├──► [2] EMA Inverter         ──► EMA features [12 dims] @200Hz
    │         (Causal Conv on MFCC)    (simulated tongue/lip/jaw positions)
    │
    └──► [3] Speaker Encoder      ──► speaker embedding [128 dims]
              (frozen WavLM CNN          (from TARGET speaker clip)
               + trainable dilated conv)
                    │
                    ▼
           [4] DDSP Vocoder        ──► Audio out (16kHz)
               (Causal Conv +
                FiLM conditioning +
                Harmonic oscillator +
                Filtered noise)
```

### Component 1: Source Extractor (pitch tracker)

- Input: mel spectrogram @200Hz
- Architecture: causal dilated convolutions (SoundStream encoder style), 3 output heads
- Outputs:
  - **f0** (fundamental frequency / pitch): classification over 256 log-spaced frequency bins, 50–500Hz
  - **periodicity**: binary voiced/unvoiced flag per frame
  - **loudness**: frame-level energy
- Training: supervised against CREPE (pitch GT) and RMVPE (voiced flag GT); noise augmented for robustness
- **Pretrained independently**, then frozen during vocoder training

### Component 2: EMA Inverter

- Input: 24-coefficient MFCC @200Hz
- Architecture: 11 dilated causal convolution layers + MLP output
- Output: 12-dim EMA vector per frame — simulated tongue tip, tongue body, lip positions etc.
- This is **acoustic-to-articulatory inversion**: predicting vocal tract geometry from sound
- Ground truth for training: **pseudo-EMA labels** generated by running the full (slow) SPARC model as a teacher
- Training: L1 loss vs. pseudo-EMA; information perturbation augmentation (pitch shift, formant shift, EQ) forces it to be speaker-agnostic
- **Pretrained independently**, then frozen during vocoder training

### Component 3: Speaker Encoder

- Input: raw waveform of TARGET speaker
- Architecture: frozen WavLM-large CNN feature extractor (@50Hz) → trainable dilated conv stack → attentive statistical pooling → MLP
- Output: 128-dim speaker embedding
- Pooling is **periodicity-weighted**: voiced frames (where timbre is clearest) contribute more to the embedding than unvoiced frames
- **Trained jointly with the vocoder**

### Component 4: DDSP Vocoder

This is the synthesis engine. DDSP = Differentiable Digital Signal Processing — rather than a black-box neural waveform model, it uses explicit audio physics baked into the architecture.

**Inputs:** f0, loudness, periodicity, EMA [all from source speech] + speaker embedding [from target speaker]

**Speaker conditioning:** FiLM layers (feature-wise linear modulation) — the speaker embedding generates scale and shift parameters that modulate every intermediate representation in the network. This is how the same articulation produces different voice timbres.

**Two synthesis branches:**

1. **Harmonic generator** (periodic / voiced sounds):  
   - Causal conv encoder maps all inputs → harmonic amplitudes (sin and cos components for 80 harmonics)  
   - Oscillators synthesize harmonics at exact integer multiples of f0  
   - Hann-window upsampling for smooth amplitude interpolation between frames  
   - Anti-alias masking removes harmonics above Nyquist  
   - Phase continuity is tracked across chunks for glitch-free streaming

2. **Noise generator** (aperiodic / unvoiced sounds like 's', 'f', 'sh'):  
   - Causal conv encoder maps inputs → 257-band frequency response H  
   - White noise is filtered by H via overlap-add (LTV-FIR filter)  
   - Overlap-add state is carried between chunks for streaming

**Output:** harmonic + noise, convolved with a speaker-specific reverb kernel (MLP from speaker embedding → 1025-tap IR)

**Training loss:** multi-scale spectral loss + multi-resolution adversarial loss (discriminator)  
**Training data:** LibriTTS-R train set — 555 hours, 2311 speakers, no target speakers seen during training

## How real-time streaming works

Every module is **causal** — it only looks at past and present, never future frames. In training, left-side padding handles the receptive field. At inference, a ring buffer stores the required past context.

The one exception: a 32ms lookahead from the STFT window (half of 1024-sample window at 16kHz). This is unavoidable and fixed.

```
Total latency = 32ms (lookahead) + 15ms (chunk size) + 14ms (processing) = 61.4ms
```

Pitch rescaling is done with a running median of source f0 (updated every chunk), so the pitch ratio `f0 * (target_median / source_median)` adapts over time without requiring the whole utterance upfront.

## Training Pipeline (the critical part — no pretrained weights exist)

Training is **three-stage** and each stage has its own data requirements:

### Stage 1: Train Source Extractor (pitch tracker)
- **Data needed:** any clean speech corpus (LibriTTS-R was used)
- **Labels:** CREPE-generated f0, RMVPE-generated voiced flags, spectral loudness
- **Label generation cost:** high — CREPE and RMVPE must be run over the entire dataset as a preprocessing step
- **Augmentation:** AddColorNoise + RoomSimulator (audiomentations library)
- **Loss:** cross-entropy (pitch bins) + cross-entropy (periodicity) + L1 (loudness)
- **Model size:** ~384-dim hidden, causal conv, lightweight

### Stage 2: Train EMA Inverter
- **Data needed:** same clean speech corpus
- **Labels (the hard part):** pseudo-EMA must be generated by running the full SPARC model (Cho et al. 2024) over the entire training set. SPARC uses a real articulatory inversion model that is slow and not in this repo.
- **Label generation cost:** very high — SPARC inference over 555 hours of speech
- **Augmentation:** information perturbation (parametric EQ + pitch randomization + formant shifting) before noise augmentation
- **Loss:** L1 vs. pseudo-EMA
- **Model size:** 384-dim hidden, causal conv

### Stage 3: Train Speaker Encoder + DDSP Vocoder jointly
- **Data needed:** same clean speech corpus, with frozen Stage 1 & 2 outputs as input features
- **No new label generation** — uses frozen pitch/EMA from stages 1 & 2
- **Loss:** multi-scale spectral loss + multi-resolution adversarial loss
- **Trainable parts:** speaker encoder (minus frozen WavLM CNN), full vocoder

**The SPARC dependency is the central constraint.** Without the SPARC model to generate pseudo-EMA labels, Stage 2 cannot be trained.

## Is this suitable for rvc-web's use case?

**Short answer: not as a drop-in replacement, and training it from scratch is a significant undertaking.**

### What it does better than RVC
| Capability | RVC | RT-VC |
|---|---|---|
| Real-time latency | ~200–500ms (depends on chunk + model) | 61.4ms |
| Zero-shot (new speaker, no fine-tune) | No — requires fine-tuning per voice | Yes |
| Noise robustness | Limited | Good (>20dB SNR) |
| Multilingual | Partial | Yes (content is language-agnostic) |
| Interpretable features | No | Yes (articulatory) |

### What makes it harder to adopt
| Issue | Impact |
|---|---|
| **No pretrained weights** | Must train everything from scratch |
| **SPARC dependency for EMA labels** | SPARC itself has limited public availability; generating 555h of pseudo-EMA labels is a multi-day preprocessing job |
| **WavLM-large dependency** | 316M parameter model needed at inference time for the speaker encoder |
| **No per-speaker fine-tuning** | Zero-shot speaker similarity (~76%) is lower than RVC fine-tuned (~85–90%); for rvc-web's use case of training a specific voice, this is a disadvantage |
| **Training infrastructure** | The paper used multi-GPU (yaml configs reference cuda:1, cuda:2, cuda:7 — likely 8-GPU setup) |
| **No training code in repo** | Repo is inference-only; training scripts, data loaders, loss functions, and the SPARC label generation pipeline are all absent |

### The core mismatch with rvc-web's use case
rvc-web's model is **per-voice fine-tuning**: a user records 5–30 minutes of their voice, trains a profile, and then converts incoming speech to sound like that specific trained voice. This is fundamentally different from RT-VC's **zero-shot** approach.

For rvc-web's use case, zero-shot is actually a weaker proposition — you trade speaker similarity quality (RVC fine-tuned produces a better match to a specific person) for the convenience of not training. If training is already part of the product, zero-shot's main advantage disappears.

### What would be worth taking from RT-VC
Even if full adoption isn't the goal, two things are directly applicable to rvc-web:

1. **The DDSP vocoder architecture** — if rvc-web's current HiFi-GAN vocoder is a latency bottleneck, the DDSP synthesis branch (harmonic + filtered noise) is ~10x faster to run than an autoregressive or GAN vocoder. It could replace the synthesis stage while keeping RVC's feature extraction and fine-tuning pipeline.

2. **The streaming/causal architecture pattern** — the left-context ring buffer approach for causal conv inference is clean and directly applicable. The current rvc-web real-time path could adopt the same chunk + context pattern.

### Realistic path if you want to use RT-VC
1. Contact the authors (UC Berkeley Speech Group) for the SPARC model and any pretrained weights — the paper is ACL 2025 so they may share on request
2. Or train only the DDSP vocoder + speaker encoder (Stage 3) using a different EMA representation — the 12 EMA dims could potentially be replaced with HuBERT soft units (which RVC already uses) at the cost of the articulatory interpretability
3. Training Stage 1 (pitch tracker) is entirely feasible without SPARC — CREPE and RMVPE are available

---

## Bottom line

RT-VC is technically impressive and the latency numbers are real. It is **not plug-and-play** — no weights, no training code, and the EMA label generation pipeline requires a separate model (SPARC) that isn't in this repo. For rvc-web's per-voice fine-tuning use case, the zero-shot framing is also a weaker fit than RVC's approach.

The most actionable piece is the **DDSP vocoder** — small, fast, and well-suited for streaming — which could be grafted onto rvc-web's existing feature pipeline as a lower-latency synthesis backend. That's a contained integration with clear payoff.
